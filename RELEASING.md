# Releasing `@slothbox/sdk`

Releases are fully automated: conventional commits drive versioning and the
changelog via [release-please](https://github.com/googleapis/release-please),
and publishing runs in GitHub Actions via
[npm trusted publishing (OIDC)](https://docs.npmjs.com/trusted-publishers)
with a [provenance attestation](https://docs.npmjs.com/generating-provenance-statements).
**There is no npm token stored anywhere** — not in repo secrets, not in org
secrets. Authentication is a short-lived OIDC token minted per workflow run.

> **Status: dormant until one-time setup is done.** The workflows are
> committed and correct, but publishing cannot succeed until the
> [one-time setup checklist](#one-time-setup-checklist) below is completed.
> Until then, do **not** merge a release PR — release-please would tag a
> release whose publish job fails.

## How a release happens, end to end

1. **Land conventional commits on `main`** (via PRs). The commit type decides
   the bump:
   - `fix:` → patch
   - `feat:` → minor
   - `feat!:` / `BREAKING CHANGE:` → minor while we are pre-1.0
     (`bump-minor-pre-major` is on), major after 1.0.0
   - `chore:`, `ci:`, `docs:`, `refactor:`, `test:` → no release on their own
2. **release-please opens/updates a release PR** (`chore: release X.Y.Z`) on
   every push to `main` (`.github/workflows/release.yml`, `release-please`
   job). The PR bumps `package.json`/`package-lock.json`, updates
   `CHANGELOG.md`, and accumulates until merged — there is no pressure to
   release every commit.
3. **A maintainer merges the release PR.** That is the human "cut a release"
   action. CI must be green like any other PR.
4. **release-please creates the tag (`vX.Y.Z`) and GitHub release** on the
   push generated by that merge.
5. **The `publish` job fires** (same workflow, gated on
   `release_created == 'true'`): checks out the tag, `npm ci`, lint, build,
   test, packaging checks, then `npm publish --provenance --access public`
   authenticated via OIDC. npm verifies the workflow identity against the
   trusted publisher configured on npmjs.com and attaches a provenance
   attestation linking the published tarball to this exact repo, tag, and
   workflow run.

The first automated release will be **v0.1.0**: the manifest
(`.release-please-manifest.json`) is bootstrapped at `0.0.0`, and the `feat:`
scaffold commits bump it to `0.1.0`, matching `package.json`.

## Who cuts releases

Anyone with write access can land conventional commits; **merging the release
PR is the release decision** and is reserved for SDK maintainers (currently
Oliver). No local commands, no manual tagging, no `npm version` — if you find
yourself typing `npm publish` outside the one-time bootstrap below, stop.

## Verifying provenance after a publish

- On npmjs.com, the package page shows a **"Provenance"** section
  ("Built and signed on GitHub Actions") linking to the source commit, the
  workflow run, and the public Sigstore transparency log entry.
- From a terminal:

  ```sh
  npm view @slothbox/sdk dist.attestations   # attestation URL + count
  npm audit signatures                        # in a consumer project: verifies
                                              # registry signatures + attestations
  ```

## Deprecating / unpublishing ("yanking") safely

Prefer, in order:

1. **Publish a fixed patch version.** Almost always the right answer.
2. **Deprecate** the bad version — consumers get a warning on install but
   nothing breaks:

   ```sh
   npm deprecate @slothbox/sdk@"0.2.3" "Broken release, use >=0.2.4"
   ```

   Deprecating a range (`@"<0.3.0"`) also works. Deprecation is reversible
   (`npm deprecate @slothbox/sdk@"0.2.3" ""`).
3. **Unpublish only as a last resort** (leaked secret in the tarball, legal
   issue). npm policy allows unconditional unpublish only within **72 hours**
   of publishing; after that, only if the package has low usage and no public
   dependents. Unpublishing breaks every consumer pinning that version, and a
   yanked version number can never be reused — the next publish must be a new
   version.

   ```sh
   npm unpublish @slothbox/sdk@0.2.3
   ```

These commands are **not** covered by OIDC (trusted publishing only covers
`npm publish`); run them logged in as an npm org member with 2FA.

## One-time setup checklist (before the first publish)

Everything below is done by a human, once. The workflows need no changes.

1. **Choose a license.** The scaffold deliberately ships without one. Add a
   `LICENSE` file and a `"license"` field to `package.json` before anything
   is published — do not publish unlicensed code.
2. **Register the `@slothbox` scope on npmjs.com** as an organization —
   done 2026-06-09. Remaining: in org settings, **require two-factor
   authentication** for all members.
3. **Make `sloth-box/sdk-typescript` public.** Required twice over: npm
   provenance is only generated for public repos, and branch protection on
   private repos needs a paid GitHub plan (see below).
4. **Publish the first version manually.** npm cannot create a brand-new
   package via OIDC ([npm/cli#8544](https://github.com/npm/cli/issues/8544));
   the package must exist before a trusted publisher can be configured. From
   a clean checkout of the release tag, as an npm org member with 2FA:

   ```sh
   npm ci && npm run lint && npm run build && npm test && npm run check:package
   npm publish --access public   # prompts for OTP
   ```

   This first version will have **no provenance attestation** (manual
   publishes can't produce one); every release after it will.
5. **Configure the trusted publisher** on
   `https://www.npmjs.com/package/@slothbox/sdk/access` (per-package, not org
   settings) → Trusted Publisher → GitHub Actions:
   - Organization or user: `sloth-box`
   - Repository: `sdk-typescript`
   - Workflow filename: `release.yml` (filename only, not the full path)
   - Environment name: leave blank (optionally create a GitHub `release`
     environment with required reviewers and add `environment: release` to
     the `publish` job for an extra approval gate)
6. **Disallow token publishing** once a trusted-publishing release has
   succeeded: package settings → Publishing access → require two-factor
   authentication and disallow tokens. This closes the stolen-token attack
   path for good.
7. **Apply branch protection on `main`** (currently blocked: the GitHub API
   returns `403 Upgrade to GitHub Pro or make this repository public` on the
   free plan while the repo is private — run this right after step 3):

   ```sh
   gh api -X PUT repos/sloth-box/sdk-typescript/branches/main/protection \
     --input - <<'JSON'
   {
     "required_status_checks": { "strict": true, "contexts": ["ci"] },
     "enforce_admins": false,
     "required_pull_request_reviews": null,
     "restrictions": null,
     "allow_force_pushes": false,
     "allow_deletions": false
   }
   JSON
   ```

   That is: the `ci` check (from `.github/workflows/ci.yml`) required and
   up to date with `main`, no force pushes, no branch deletion.

Already verified (2026-06-09), no action needed: repo Actions settings allow
workflows to create pull requests (release-please needs this), and the
default `GITHUB_TOKEN` workflow permissions are sufficient.

## Notes

- **npm CLI requirement:** trusted publishing needs npm ≥ 11.5.1 (node ≥
  22.14). The publish job uses node 24 and explicitly runs
  `npm install -g npm@latest` so it never depends on the runner image's
  bundled npm.
- **PyPI next:** `sdk-python` will reuse this exact pattern with PyPI trusted
  publishing. One difference to remember: PyPI supports *pending* publishers,
  so the trusted publisher can be configured **before** the package exists —
  no manual first publish there.
